T‑Ивент

Frontend Backend QA

Проблема

Поездка или общее мероприятие — десятки трат, разные плательщики. Кто за кого платил и кто кому должен — считается вручную, с ошибками и спорами.

Решение — T‑Ивент

  • Фиксируем расходы в одном месте: кто платил, за кого, сколько
  • Система сама делит поровну и считает взаиморасчёты
  • Прозрачно: «кто кому сколько должен» — без ручных подсчётов

Frontend

PWA на React, типобезопасность, FSD-структура

Frontend — стек

TypeScript 5.7 React 19 Vite Tailwind CSS React Query react-hook-form Zod Zustand Собственный UI-kit Vitest Playwright

Особенности

  • Собственный UI-kit: переиспользуемые компоненты (формы, модалки, кнопки) — единый визуальный стиль
  • Feature-Sliced Design: entities / features / widgets / pages — масштабируемая структура
  • Валидация форм Zod + react-hook-form — единый источник правды по схеме
  • React Query: кэш, инвалидация, авто-рефетч после мутаций
  • Авто-refresh JWT в axios-интерсепторе (прозрачно для пользователя)

Backend

Spring Boot, модульная архитектура, транзакции

Backend — стек

Java 25 Spring Boot 4 Spring Modulith Maven PostgreSQL Redis Liquibase OpenAPI / springdoc jjwt 0.12

Особенности

  • Разграничение доступа: операции владельца проверяются на сервере (AccessDenied)
  • Транзакции БД при пересчёте балансов — защита от рассинхрона
  • Liquibase-миграции, версионируемая схема
  • Redis — кэш / служебные данные (в т.ч. контроль orphan-загрузок S3)

Расчёт долгов и взаиморасчётов

  • Расход делится поровну между плательщиком и выбранными участниками (HALF_UP, 2 знака)
  • Пересчёт сальдо автоматически при создании и изменении расхода
  • Взаиморасчёты — алгоритм минимизации числа переводов (min-cash-flow): не «все платят всем», а минимум транзакций

Пример: расход 3000 ₽ на 3 человек → доля каждого 1000 ₽, баланс пересчитывается без ручного вмешательства.

Инфраструктурные фишки

CI/CD на GitHub Actions

Отдельные workflow-файлы, запускаются автоматически на push / PR:

check

Сборка и проверка прод-кода → Docker build, push в GHCR

e2e

Подъём backend + frontend → Playwright e2e

api

Подъём backend → API-тесты

deploy

Доставка собранного образа на стенд

Деплой собирает приложение; e2e/api/unit — отдельные проверки, не блокируют сборку образа.

S3-хранилище изображений

  • MinIO (S3-совместимое) в Docker, отдельный контейнер + авто-создание бакета
  • Presigned URL: файл грузится напрямую в хранилище, не через backend — масштабируемо, не нагружает приложение
  • Изоляция по пользователю: ключ receipts/{userId}/..., проверка владения
  • Лимит размера файла, очистка «осиротевших» загрузок через Redis
  • Применяется для: аватар профиля, обложка события, чек расхода

Сильные стороны проекта

Безопасность

JWT access + refresh, защищённые маршруты, server-side проверка прав и редирект при 403.

Архитектура

FSD на frontend, доменное разделение backend, миграции Liquibase и явный API-контракт.

Расчёт взаиморасчётов

Автоматический пересчёт долей и алгоритм минимизации переводов вместо ручных таблиц.

Качество

Unit, e2e и API-тесты покрывают критичные сценарии и снижают риск регрессий при изменениях.

Итог

T‑Ивент закрывает полный сценарий совместных расходов:

  • Пользователь создаёт событие, приглашает участников и фиксирует расходы в одном месте
  • Участники подтверждают траты, а система автоматически считает доли и взаиморасчёты
  • Доступ защищён: JWT-сессия, приватные маршруты и серверная проверка прав
  • Проект можно развивать дальше: архитектура разделена по доменам, API описан, критичные сценарии покрыты тестами
