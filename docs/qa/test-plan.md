# Test.Plan.V5 | T-Event | 11.04.2026

## 1. Цель документа
Описать подход к тестированию проекта **T-Ивент** — веб-приложения для совместного управления бюджетом события.

Цель тестирования — проверить, что ключевые пользовательские сценарии, API и бизнес-логика работают корректно, выявить критические ошибки до демонтсрации/сдачи проекта.

## 2. Область тестирования
### Покрытие:
- регистрация и авторизация
- создание события / поездки
- приглашение по ссылке
- подтверждение участия
- роли владельца и участника
- создание, просмотр, редактирование расходов
- равное деление расходов
- пересчёт баланса после изменения расходов
- отображение итогов: кто кому сколько должен
- история действий
- экспорт итогов в CSV
- базовые проверки доступа и ограничений прав

### Вне покрытия:
- кроссбраузерность сверх заявленных 
- защита от сложных конкурентных сценариев, отказоустойчивость под высокой нагрузкой

## 3. Приоритеты тестирования

## 4. Подход к тестированию
### Виды тестирования
В проекте используются следующие виды тестирования:
- **Sanity testing**: точечная проверка конкретного исправления или небольшой доработки
- **Smoke testing**: быстрая проверка критических сценариев после нового билда, перед демонстрацией и сдачей
- **Functional testing**: проверка пользовательских и бизнес-сценариев
- **API testing**: проверка эндпоинтов, кодов ответа, валидации, авторизации
- **UI testing**: проверка поведения интерфейса
- **Regression testing**: перепроверка ранее работающего функционала после изменений
- **Negative testing**: проверка невалидных данных, некорректных прав доступа, ошибочных действий пользователя
- **Integration testing**: проверка взаимодействия между компонентами системы

### Пирамида тестирования
Подход включает в себя:
- Unit: покрытие отдельных функций и компонентов (в основном зона ответственности разработчиков)
- Integration: проверка взаимодействия API, БД и бизнес-логики
- E2E: проверка критических пользовательских сценариев через UI

### Тестовая стратегия

Основной упор в проекте делается на функциональное, API- и UI-тестирование, а также на smoke- и regression-проверки критических сценариев.

#### Покрытие API
Для API проверяются:
- корректные запросы
- валидация входных данных
- коды ответа
- ошибки авторизации
- доступ к чужим данным
- обработка некорректных входных значений

#### Покрытие UI
Через UI проверяются основные пользовательские сценарии:
- регистрация
- вход
- создание события
- присоединение по ссылке
- добавление расхода
- просмотр итогов

#### Покрытие бизнес-логики
Особое внимание уделяется:
- расчёту долей
- пересчёту баланса после изменения расхода
- корректному отображению итоговых взаиморасчётов

#### Регрессионные проверки
Регрессионное тестирование выполняется перед демонстрацией и после исправления критичных дефектов.

## 5. Окружение тестирования
- frontend: local / dev
- backend: local / dev
- database: PostgreSQL
- browsers: Chrome, Firefox
- adaptive: mobile + desktop
- OS: Windows, Linux

## 6. Тестовые данные
Для тестирования должны быть подготовлены:
- минимум 2–3 тестовых пользователя
- минимум 1 владелец события
- минимум 2 участника события
- события с разным количеством участников
- расходы с разными суммами
- категории по умолчанию и пользовательские категории
- сценарии с редактированием расходов
- сценарии с повторным входом по invite link
- невалидные суммы / пустые поля / некорректные значения
- сценарии с доступом пользователя к чужому событию

## 7. Подход к автоматизации
 - Postman используется для ручной проверки API, подготовки коллекции запросов и базовых проверок ответов
 - Jest планируется использовать для автоматизации API-тестов, включая позитивные и негативные сценарии
 - Playwright используется для E2E и smoke-проверок критических пользовательских сценариев
 - В первую очередь автоматизируются критичные сценарии, влияющие на демо и основные бизнес-потоки: авторизация, создание события, присоединение по ссылке, добавление расхода, просмотр итогов

## 8. Работа с дефектами
### Баг-репорт содрежит:
- заголовок
- окружение
- предусловия
- шаги воспроизведения
- ожидаемый результат
- фактический результат
- severity
- вложения

### Баг-трекер
- Баги фиксируются в GitHub Projects / QA-issues доска команды

### Проверка исправлений:
- после исправления баг повторно проверяется QA
- статусы процессов работы с дефектом на доске new/in-progress/fixed

## 9. Риски
### Технические:
- сложная логика расчётов
- нестабильный API
### Процессные:
- изменение требований по ходу работы
- несогласованность frontend/backend контрактов
### Ресурсные:
- 1 QA на проект
- ограниченное время на проведение полного регрессионного тестирования
### Коммуникационные:
- задержки в исправлении дефектов из-за ограниченного времени команды

## 10. Критерии старта/завершения тестирования
### Тестирование может быть начато, если:
- есть доступный frontend и backend, билд запускается локально
- согласован API-контракт
- определен список реализованного функционала на текущий этап
- доступны тестовые данные / тестовые аккаунты

### Тестирование может быть закончено, если:
- протестированы все критичные сценарии
- нет открытых блокеров / критичных багов
- критические дефекты закрыты или согласованы как известные ограничения
- подготовлены основные QA-артефакты
- результаты тестирования зафиксированы
- smoke для критических сценариев выполнен успешно
- regression для затронутого функционала выполнен

## 11. Роли и ответственность
- QA: подготовка тестовой документации, выполнение ручного тестирования, баг-репорты, smoke/regression тестирование
- Backend: исправление дефектов API, бизнес-логики и серверной части
- Frontend: исправление дефектов UI, клиентской логики и пользовательских сценариев
- Команда: согласование требований, приоритетов и статуса исправлений дефектов
