# Test Plan

## 1. Purpose
План тестирования описывает подход к проверке веб-приложения T-Ивент, предназначенного для совместного управления бюджетом события или поездки.
Цель тестирования - подтвердить, что ключевые пользовательские сценарии работают корректно, расчёты выполняются без ошибок, а данные участников и событий остаются целостными и доступны только авторизованным пользователям.

## 2. Test Object
Объект тестирования:
- frontend SPA на React + TypeScript
- backend API на Spring Boot
- база данных PostgreSQL
- пользовательские сценарии управления событиями, участниками, расходами и взаиморасчётами

## 3. Scope
В рамках тестирования проверяются:
- регистрация и авторизация пользователей
- создание события или поездки
- приглашение и подтверждение участия
- роли владельца и участника
- создание и редактирование категорий бюджета
- добавление, изменение и удаление расходов
- распределение расходов между участниками
- автоматический пересчёт долей и баланса
- отображение итогов: кто кому сколько должен
- история действий по событию
- экспорт итогов в CSV

Не входят в текущий scope или проверяются ограниченно:
- нагрузочное тестирование
- полноценное security testing beyond базовые проверки авторизации и прав доступа
- кроссбраузерное тестирование на широком наборе браузеров
- мобильные приложения, так как продукт является веб-приложением

## 4. Test Goals
Основные цели тестирования:
- подтвердить выполнение обязательных бизнес-требований из README
- проверить корректность критичных финансовых расчётов
- убедиться, что пользователь не может получить доступ к чужим событиям и данным без прав
- убедиться, что история изменений и итоговые данные отображаются корректно
- выявить дефекты до демонстрации, сдачи или релиза

## 5. Test Types
Планируется использовать следующие виды тестирования:
- smoke testing
- functional testing
- regression testing
- API testing
- UI testing
- exploratory testing
- negative testing
- basic authorization and access testing

Дополнительно по мере развития проекта:
- e2e testing для ключевых пользовательских сценариев
- проверка CSV export
- проверка корректности обработки ошибок API и UI

## 6. Test Levels
### 6.1 Unit Tests
Проверка отдельных бизнес-методов и вычислений на backend, особенно:
- расчёт долей
- пересчёт сальдо
- валидация данных
- права доступа на уровне сервисов

### 6.2 Integration Tests
Проверка взаимодействия:
- backend + database
- backend API + business logic
- persistence layer и транзакционности операций

### 6.3 System / E2E Tests
Проверка пользовательских сценариев через UI или через связку frontend + backend:
- вход пользователя
- создание события
- приглашение участника
- добавление расхода
- просмотр итоговых взаиморасчётов

## 7. Priority Areas
Наивысший приоритет тестирования:
- авторизация и разграничение доступа
- создание события
- приглашение участников
- добавление расходов
- корректность расчётов "кто кому сколько должен"
- пересчёт баланса после изменения расхода

Средний приоритет:
- категории бюджета
- история действий
- экспорт в CSV

Низкий приоритет:
- второстепенные UI-состояния
- текстовые и визуальные неточности, не влияющие на основной пользовательский сценарий

## 8. Main Test Scenarios
### 8.1 Authentication
- успешная регистрация нового пользователя
- успешная авторизация
- авторизация с неверным логином или паролем
- валидация обязательных полей
- доступ к защищённым разделам только для авторизованного пользователя

### 8.2 Event Creation and Management
- создание события с валидными данными
- проверка обязательных полей при создании события
- редактирование события владельцем
- запрет на редактирование события участником без нужных прав

### 8.3 Invitations and Participation
- приглашение участника по ссылке
- приглашение участника по e-mail, если реализовано
- подтверждение участия
- защита от повторного или некорректного приглашения

### 8.4 Expense Management
- добавление расхода с валидными данными
- добавление расхода с разными плательщиками и составом участников
- редактирование расхода
- удаление расхода
- валидация суммы, категории и обязательных полей
- обработка дробных значений и округления

### 8.5 Balance Calculation
- корректный расчёт долей между участниками
- пересчёт после добавления расхода
- пересчёт после изменения расхода
- пересчёт после удаления расхода
- отображение итогового баланса по событию

### 8.6 History
- запись действий в ленту истории
- корректное отображение типа события, автора и времени действия

### 8.7 Export
- успешный экспорт итогов в CSV
- корректность структуры CSV-файла
- корректность данных в экспортируемом файле

## 9. Test Environment
Минимальное тестовое окружение:
- frontend: локальный dev-стенд
- backend: локальный или dev API
- database: PostgreSQL 16
- browser: Google Chrome latest stable
- OS: Linux

Желательно дополнительно проверять:
- Firefox latest stable
- сценарии с "чистой" сессией браузера
- сценарии с несколькими пользователями

## 10. Test Data
Для тестирования потребуются:
- минимум 3 тестовых пользователя
- минимум 2 события с разным составом участников
- расходы с целыми и дробными суммами
- данные с валидными и невалидными значениями
- категории по умолчанию и пользовательские категории

## 11. Entry Criteria
Тестирование может быть начато, если:
- приложение запускается локально или на dev-стенде
- backend доступен и подключён к БД
- основные пользовательские сценарии реализованы
- тестовые данные подготовлены или могут быть созданы через UI/API

## 12. Exit Criteria
Тестирование может считаться завершённым, если:
- выполнены все smoke-сценарии
- пройдены все критичные сценарии из scope
- для найденных дефектов созданы bug reports
- отсутствуют открытые блокирующие и критические дефекты для релиза или демонстрации

## 13. Deliverables
Артефакты тестирования:
- test plan
- checklist / test cases
- bug reports
- Postman collection for API checks
- smoke test results
- regression results before demo or release

## 14. Risks
Основные риски:
- отсутствие e2e-инфраструктуры на текущем этапе
- неполная стабилизация API и UI параллельно с разработкой
- ошибки округления и финансовых расчётов
- недостаточное покрытие сценариев прав доступа
- расхождение поведения frontend и backend валидации

## 15. QA Approach for Current Stage
На текущем этапе рекомендуется следующий практический подход:
- вручную покрыть smoke и critical path сценарии
- подготовить bug report template и примеры оформления багов
- собрать минимальный regression checklist
- подготовить Postman-коллекцию для backend API
- позже добавить e2e smoke для ключевых сценариев: login, create event, add expense, view balance
