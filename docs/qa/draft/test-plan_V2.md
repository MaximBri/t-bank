# Test Plan V2

## 1. Введение

- Проект T-Ивент — веб-приложение для совместного управления бюджетом события или поездки.

- Пользователи могут создавать событие, приглашать участников по ссылке, добавлять расходы, делить суммы между участниками и смотреть итоговые взаиморасчёты.

- Цель тестирования — проверить корректность ключевой бизнес-логики, стабильность основных пользовательских сценариев, корректность API и UI, а также снизить риск критических дефектов перед демо.

## 2. Область тестирования

> В покрытие входят:

- регистрация и авторизация
- создание события
- приглашение участника по ссылке
- подтверждение участия
- роли владелец/участник
- создание и редактирование расходов
- равное деление расходов между выбранными участниками
- пересчёт баланса и отображение "кто кому сколько должен"
- история действий
- экспорт итогов в CSV
- базовая проверка ограничений доступа

- Вне покрытия или в низком приоритете на первом этапе:

- нефункциональное нагрузочное тестирование
- глубокая проверка accessibility
- локализация
- email-интеграции
- защита от сложных конкурентных сценариев на уровне production-like нагрузки, если они не реализованы в MVP

## 3. Тестовая стратегия

- Основной упор делается на:

- функциональное тестирование
- API-тестирование
- UI-тестирование
- smoke/regression проверки

- Для API проверяются:

- корректные запросы
- валидация
- коды ответов
- ошибки авторизации
- доступ к чужим данным
- обработка некорректных входных значений

- Для UI проверяются основные пользовательские потоки:

- регистрация
- вход
- создание события
- присоединение по ссылке
- добавление расхода
- просмотр итогов

- Для бизнес-логики особый приоритет имеют:

- расчёт долей
- пересчёт баланса после изменения расхода
- отображение итоговых взаиморасчётов

- Регрессионное тестирование выполняется перед демо и после исправления критичных багов.

## 4. Окружения

- Основное окружение тестирования — локальный запуск frontend + backend + database.

- Дополнительно используется Postman для API-проверок и тестовая база данных с подготовленными пользователями и событиями.

- Если будет staging-окружение, на нём выполняются smoke и итоговые regression-проверки перед демонстрацией.

- Для UI желательно проверить приложение:

- минимум в актуальном Chrome
- дополнительно в Firefox, если позволит время
- мобильную адаптацию через DevTools

## 5. Подход к автоматизации

- Для API автоматизации приоритетны критические эндпоинты:

- авторизация
- создание события
- приглашение / подтверждение участия
- добавление расхода
- получение итоговых взаиморасчётов

- Инструменты:

- Postman collection
- и/или Jest для API-проверок, если команда пойдёт по JS/TS стеку QA

- Для UI автоматизации приоритетны smoke-сценарии через Playwright:

- вход пользователя
- создание события
- присоединение по ссылке
- добавление расхода
- проверка итогов

- В первую очередь автоматизируются стабильные и повторяемые сценарии.

- Ручными остаются exploratory и edge-case проверки.

## 6. Баг-репорты

- Все дефекты фиксируются в согласованном формате с обязательными полями:

- заголовок
- окружение
- предусловия
- шаги воспроизведения
- ожидаемый результат
- фактический результат
- severity
- priority
- вложения при необходимости

- Баги хранятся в общем трекере команды или в отдельном markdown/таблице, если трекера нет.

- Критичные дефекты:

- невозможность логина
- невозможность создания события
- невозможность добавления расхода
- неправильный расчёт долей
- доступ к чужим данным
- ошибки 500 в штатных сценариях

- После исправления баг обязательно проходит retest и при необходимости regression смежного функционала.

## 7. Риски и ограничения

- Основные риски:

- несогласованность frontend и backend контракта
- изменение требований по ходу проекта
- отсутствие стабильного staging-окружения
- нехватка тестовых данных
- ошибки в расчётах при изменении расходов
- неполная обработка негативных сценариев авторизации и прав доступа
- ограниченное время на полную regression-проверку

- Ограничения:

- маленькая команда
- учебный формат
- сжатые дедлайны
- возможное отсутствие полноценного bug tracker и CI для автотестов на ранних этапах

- Приоритет тестирования в условиях ограниченного времени — критические бизнес-сценарии и smoke flow.
